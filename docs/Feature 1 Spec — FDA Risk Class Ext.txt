Feature 1 Spec — FDA Risk Class Extraction and Display (v2, calibrated)
Context and goal
AdvisoryOps currently surfaces 873 healthcare-relevant issues but 77% of them are P3 (informational). The FDA regulatory risk classification of a medical device (Class I, II, or III) is the single most important signal for distinguishing "life-critical pacemaker vulnerability" from "disposable supply recall," and it's currently not used anywhere in the pipeline. Adding it as both a visible dashboard field and a scoring signal will elevate Class III issues out of the P3 noise into P2/P1 visibility where analysts actually see them.
Data sources
Primary source: The openFDA recall cache at outputs/openfda_cache/recall_*.json. Each cached recall record contains a device_class field with values "1", "2", "3", or occasionally "N" / "U" / missing. Sample analysis of 2000 records shows:

Class 1: 17.6%
Class 2: 72.0%
Class 3: 6.5%
Missing/invalid: ~3.7% (N, U, or absent)

Both recall.device_class (top-level) and recall.openfda.device_class (nested) must be checked. 99.2% of records have a product_code, 100% have a device_name.
Secondary source: openFDA Device Classification endpoint at https://api.fda.gov/device/classification.json. This is a separate database of ~6,000 device classification records keyed by product code. Used as a fallback when an issue mentions a medical device but isn't derived from a recall.

Add a function to src/advisoryops/enrichment/ (create a new module fda_classification.py if one doesn't exist). Function signature:

Implementation:

Check recall.get("device_class") first
Fall back to recall.get("openfda", {}).get("device_class")
If the value is a list, take the first element
Coerce to string
If the result is one of "1", "2", "3": return it
Otherwise return None (do not map "N" or "U" to anything)

Step 4: Wire extraction and lookup into the pipeline
Find the pipeline stage where issues are enriched after correlate but before scoring. This is likely src/advisoryops/enrichment/ or called from community_build.py.
For each issue:

If the issue has a source indicating openFDA recall origin, look up the original cached recall record and call extract_risk_class_from_recall(). If it returns a value, set issue["fda_risk_class"].
If step 1 didn't set the field, attempt a secondary lookup via lookup_risk_class() using any product code or device name mentioned in the issue. If a match is found with reasonable confidence, set issue["fda_risk_class"].
If neither works, leave fda_risk_class as null.

The secondary lookup should be gated by device name keyword matching. Don't run it on every issue — only on issues where device-like keywords appear in the title or summary. Otherwise we'll burn cycles on CVE-2024-XXXXX Chrome browser issues that have nothing to do with medical devices.
Step 5: Add risk class to the scoring system
Modify src/advisoryops/score.py to add a new dimension. The bonus structure based on calibration against the real 873-issue healthcare corpus:
pythondef _score_fda_risk_class(issue: dict) -> tuple[int, list[str]]:
    """Calculate FDA risk class bonus.
    
    Calibrated against real healthcare corpus distribution:
    - Class III: +30 (promotes critical devices from P3 to P2, P2 to P1)
    - Class II: +10 (modest nudge, Class II is 72% of FDA recalls)
    - Class I:  +0  (genuinely low-risk devices, no bonus)
    - null:     +0  (don't fake certainty)
    """
    rc = issue.get("fda_risk_class")
    if rc == "3":
        return 30, ["fda-risk-class: Class III highest-risk device (+30)"]
    if rc == "2":
        return 10, ["fda-risk-class: Class II moderate-risk device (+10)"]
    return 0, []
Add the call to score_issue_v2() after the clinical signals dimension and before the final total. Append the reasoning strings to the why list. Make sure the bonus is documented in the theoretical max table in docs/scoring_internals.md.

Edge cases

"N" and "U" device_class values: Appear in ~2% of records. Reject them as invalid. Do not map to any risk class.
Recall records with device_class as an integer: Coerce to string before comparison.
Issues derived from a recall that's in the old cache but not in the new normalized feed: Check that the cache-to-feed mapping is traceable. If an issue's source points to a recall but we can't find the original record, leave risk class null and don't crash.
Duplicate product codes in the classification database: Unlikely but possible. If encountered, use the first match.
Classification database refresh: Cache is considered stale after 90 days. Refresh in the background; don't block the pipeline if refresh fails.
Dashboard issue cards with null risk class: Do not display any badge. Do not display "Class Unknown" or "N/A." Silent omission is correct.

Success criteria

All unit tests pass (integration test optional but should pass when run with -m integration)
Pipeline regenerates and feed_healthcare.json contains fda_risk_class populated on at least 40% of issues (based on 45% of healthcare issues having openFDA recall origins in the current corpus)
Spot check: pick 10 random Class III issues from the regenerated feed and verify they look clinically significant (pacemakers, implantable defibrillators, ventricular assist devices, etc.). Pick 10 random Class I issues and verify they look low-risk.
Score distribution shift: before/after comparison shows Class III issues have meaningfully higher scores (average +25-30 points). Class II issues shift modestly (+8-10). Total number of P0/P1/P2 healthcare issues increases by approximately 50-100 as Class III items promote up from P3.
Dashboard visibly shows Class III badges in red on appropriate issues
Filtering by "Class III only" returns only Class III issues
docs/scoring_internals.md is updated to include the new risk class dimension in the theoretical max table

What this feature does NOT include

No LLM involvement (this is pure data extraction, lookup, and scoring)
No citation verification framework (comes in Feature 3)
No changes to the healthcare filter itself (separate concern — the healthcare_category field being empty is a known bug but not fixed here)
No Excel export fix (separate low-priority bug)
No changes to NVD historical cache integration (separate concern)
